당신은 이 프로젝트를 처음부터 끝까지 책임지는 자율적인 시니어 모바일 엔지니어, QA 엔지니어, 기술 리드다.

현재 저장소에서 Expo React Native 기반의 크로스플랫폼 MVP를 완성하라.

단순히 코드를 생성하거나 일부 화면만 만든 뒤 멈추지 마라. 저장소 조사, 설계, 구현, 테스트 작성, Android 에뮬레이터 실행, 실제 사용자 흐름 검증, 로그 확인, 버그 수정, 재검증, 문서화까지 스스로 반복하라.

최종 완료 조건을 모두 충족하거나, 현재 환경에서 기술적으로 실행할 수 없는 외부 차단 요인을 정확한 명령 출력과 함께 입증하기 전에는 작업을 끝내지 마라.

사용자에게 중간 승인이나 사소한 결정을 묻지 마라. 명세에 없는 작은 사항은 아래 우선순위에 따라 합리적으로 결정하라.

1. 기능의 정확성과 안정성
2. 재현 가능한 테스트
3. Android 에뮬레이터에서의 실제 작동
4. Expo 크로스플랫폼 호환성
5. 구현 단순성
6. 개발 속도
7. 시각적 완성도

UI 미관은 중요하지 않다. 플레이스홀더 수준의 단순한 UI로 만들되, 모든 핵심 기능과 상태 전환은 온전히 작동해야 한다.

- Expo Router를 사용한다.

- TypeScript strict mode를 유지한다.
- `any` 사용을 피한다.
- 데이터와 핵심 로직을 화면 컴포넌트에 직접 섞지 않는다.
- 실제 회원가입, 실제 결제, 실제 소셜 백엔드, 실제 제휴 정산은 만들지 않는다.

- API 실패가 앱 전체 실패로 이어져서는 안 된다.
- 비밀 키를 모바일 번들, `EXPO_PUBLIC_*`, 소스코드, Git에 절대 넣지 않는다.
- EWG나 다른 웹사이트를 스크래핑하지 않는다.
- “유해도”, “안전 보장”, “진단”, “치료”, “트러블 확률” 같은 표현을 사용하지 않는다.
- 피부 질환을 진단하거나 치료를 권유하지 않는다.
- 피부 사진 분석은 구현하지 않는다.
- 모든 분석 화면에 참고 정보이며 의학적 평가가 아니라는 고지를 표시한다.
- 프로젝트 외부로 푸시하거나 실제 비용이 발생하는 작업은 하지 않는다.
- 실제 외부 구매 링크는 열지 않는다.
- 보너스 기능은 구현하지 말고 `FUTURE_ROADMAP.md`에만 기록한다.
- P0 기능을 모두 검증하기 전에는 보너스 기능을 건드리지 않는다.
- 성공했다고 보고하기 전에 반드시 Android 에뮬레이터 검증 증거를 남긴다.
- 에뮬레이터를 실행하지 못했다면 “Android 검증 완료”라고 주장하지 않는다.

스캔 화면은 다음 세 경로를 지원해야 한다.

D. 샘플 제품으로 빠른 테스트

실제 성분 분석을 하려면 다음 중 하나가 필요하다.

- 사용자가 전성분을 직접 입력함
- 사전 준비된 샘플 제품을 선택함

제품 앞면 사진만 있고 전성분 정보가 없으면:

- 제품명 초안만 저장 가능
- 성분 분석 버튼은 비활성화
- “정확한 분석을 위해 전성분표 사진이나 직접 입력이 필요해요” 표시

- 카메라 취소 처리

6. 분석 모드
==================================================

분석기는 인터페이스로 분리한다.

예시:

- `ProductAnalyzer`
- `MockProductAnalyzer`
- `RemoteProductAnalyzer`

기본값:

- API 설정이 없으면 `MockProductAnalyzer`
- 앱 전체가 데모 모드에서 완전히 작동
- 데모 결과를 실제 AI 결과처럼 속이지 않는다.
- 화면에 `데모 분석 결과` 배지를 표시

환경변수:

- `EXPO_PUBLIC_ANALYZER_MODE=mock|remote`
- `EXPO_PUBLIC_ANALYSIS_API_URL=http://10.0.2.2:8787`
- `GEMINI_API_KEY=`
- `GEMINI_MODEL=gemini-2.5-flash`
- `ANALYZER_PORT=8787`

Android 에뮬레이터에서 로컬 프록시를 사용할 때 기본 API URL은 `http://10.0.2.2:8787`을 사용할 수 있다.

실제 iPhone에서 Expo Go로 실행할 경우 `10.0.2.2`는 동작하지 않는다. 이 경우 `EXPO_PUBLIC_ANALYSIS_API_URL`은 개발자 컴퓨터의 LAN IP 주소 또는 tunnel/ngrok URL을 사용해야 한다.

예:

- Android emulator: `http://10.0.2.2:8787`
- Physical iPhone: `http://192.168.x.x:8787` 또는 tunnel URL

중요:

- `GEMINI_API_KEY`는 모바일 앱 코드에서 절대 참조하지 않는다.
- `GEMINI_API_KEY`는 `EXPO_PUBLIC_` 접두사를 사용하지 않는다.
- `.env`를 Git에 넣지 않는다.
- `.env.example`에는 빈 값만 제공한다.

선택적 실제 AI 기능을 위해 저장소 안에 가벼운 로컬 프록시를 만든다.

예시:

- `server/index.ts`
- `GET /health`
- `POST /analyze-product`

프록시 요구사항:

- Node.js TypeScript
- `@google/genai`
- Zod 검증
- API 키는 서버 환경변수에서만 읽음
- 기본 모델은 환경변수로 교체 가능
- Gemini 구조화 출력 사용
- 이미지와 성분 원문을 로그에 출력하지 않음
- 요청 크기 제한
- 30초 타임아웃
- 잘못된 응답 스키마 처리
- API 키 없음은 명확한 503 오류 코드
- 모바일 앱은 이 오류를 잡아 직접 입력 또는 데모 분석을 제안
- API 실패 시 사용자가 입력한 내용이나 선택한 사진을 잃지 않음
- 실제 Gemini 호출은 API 키가 있을 때만 테스트
- API 키가 없다는 이유로 전체 검증을 실패 처리하지 않음

프록시에 전달할 AI 지시:

9. 제품 조합 규칙
==================================================

다음 규칙을 코드로 구현하고 단위 테스트한다.

1. 레티노이드와 AHA/BHA/PHA 제품을 같은 저녁 루틴에 동시에 넣지 않는다.
2. 함께 보유하고 있으면 “민감 피부는 번갈아 사용 참고”를 표시한다.
3. 각질 관리 제품이 두 개 이상 같은 루틴에 선택되면 하나만 선택한다.
4. Retinol과 Retinal은 기본적으로 PM에만 배치한다.
5. Sunscreen은 AM 마지막 단계에만 배치한다.
6. Cleanser는 각 루틴의 첫 단계다.
7. Moisturizer는 Sunscreen 전 또는 PM 마지막 보습 단계다.
8. Oil은 PM에서 Moisturizer 다음 단계로 둘 수 있다.
9. 높은 민감도와 향료 또는 에센셜 오일 플래그가 함께 있으면 참고 경고를 만든다.
10. 사용자가 피하고 싶은 성분과 정확히 또는 별칭으로 일치하면 강한 개인 설정 경고를 만든다.
11. 이 규칙을 의학적 금기라고 표현하지 않는다.
12. “주의”, “번갈아 사용 참고”, “패치 테스트 고려” 수준으로 표현한다.

11. 화장대 구성 점수
==================================================

점수 이름:

“화장대 구성 점수”

항상 다음 안내를 붙인다.

“제품 구성의 균형을 이해하기 위한 참고 점수이며 피부 건강이나 의학적 상태를 평가하지 않아요.”

결정론적 점수 공식을 구현한다.

총점 100점:

A. 기본 구성 충족: 40점

- Cleanser 존재: 8점
- Moisturizer 또는 Barrier 기능 제품 존재: 16점
- Sunscreen 존재: 16점

B. 사용자 고민 기능 충족: 30점

- 사용자가 선택한 고민 하나당 최대 15점
- 해당 고민에 대응하는 기능을 가진 제품이 있으면 15점
- 고민이 한 개만 있으면 해당 항목 최대 30점으로 정규화하거나 계산 이유를 명확히 한다.

C. 조합 관리: 15점

- 15점에서 시작
- 고유한 높은 주의 조합 한 개당 5점 감점
- 최소 0점

D. 중복 관리: 15점

- 15점에서 시작
- 동일 카테고리와 주요 기능이 겹치는 제품이 2개를 초과할 때 초과 제품당 3점 감점
- 최소 0점

최종 점수:

- 반올림
- 0~100 범위
- 제품이 없으면 0점

화면에는 총점만 보여주지 말고 반드시 계산 근거를 보여준다.

표시:

- 기본 구성
- 고민 기능 커버리지
- 발견된 중복
- 조합 참고
- 부족한 기능
- 가장 중요한 다음 행동 한 개

특정 제품 구매를 무조건 권하지 않는다.

예:

“현재 화장대에는 자외선 차단 단계가 없어요.”

“보습 세럼 역할이 겹치는 제품이 세 개 있어 새 세럼의 구매 우선순위는 낮아요.”

12. AM/PM 기본 루틴
==================================================

매일 피부 상태를 입력하게 하지 않는다.

AM 기본 순서:

1. cleanser
2. toner
3. essence
4. serum
5. treatment
6. moisturizer
7. sunscreen

PM 기본 순서:

1. cleanser
2. toner
3. essence
4. serum
5. treatment
6. moisturizer
7. oil

규칙:

- 같은 단계의 모든 제품을 무조건 넣지 않는다.
- 주요 기능이 겹치면 대표 제품만 선택하고 나머지는 대체 제품으로 표시한다.
- 레티노이드와 각질 제품은 같은 PM 루틴에 함께 넣지 않는다.
- 충돌하는 제품은 “번갈아 사용 참고” 영역으로 분리한다.
- 레티노이드는 PM 우선
- Sunscreen은 AM만
- 각 제품마다 선택 이유 한 줄 표시
- 제품이 부족해도 오류를 내지 않고 현재 가능한 루틴을 생성
- 루틴이 비어 있으면 제품 추가 CTA 표시
- 루틴 생성 알고리즘은 순수 함수로 작성하고 단위 테스트

- 확인 필요 제품에는 구매 CTA를 강조하지 않음

23. Android 에뮬레이터 검증
==================================================

Android를 1차 필수 검증 대상으로 삼는다. iOS 시뮬레이터나 iPhone 검증은 필수 완료 조건이 아니지만, 구현 중 Android 전용 API 사용을 피하고 Expo Go에서 iPhone 실행이 가능한지 README에 실행 방법과 주의사항을 문서화한다.

저장소 작업을 시작하기 전에 환경을 조사한다.

확인:

- `node --version`
- `npm --version`
- `adb --version`
- `adb devices`
- `emulator -list-avds`
- Android SDK 경로
- 사용 가능한 AVD
- Expo MCP 도구 사용 가능 여부

에뮬레이터가 이미 있으면 가장 안정적인 Pixel 계열 AVD를 사용한다.

에뮬레이터가 없고 설치된 시스템 이미지가 있다면 새 AVD를 생성해도 된다.

Android SDK나 시스템 이미지 자체가 없다면:

- 코드와 자동 테스트는 끝까지 완료
- 설치에 필요한 정확한 차단 요인과 명령을 보고
- 에뮬레이터 검증을 했다고 주장하지 않음

실행 우선순위:

1. Expo Go와 `npx expo start --android`
2. Expo MCP 로컬 자동화
3. 필요할 때만 `npx expo run:android`
4. custom native build는 Expo Go로 불가능한 기능이 있을 때만

Expo MCP가 연결되어 있으면:

- `npx expo install expo-mcp --dev`
- 같은 Expo 계정 로그인 확인
- `EXPO_UNSTABLE_MCP_SERVER=1 npx expo start`
- Android 에뮬레이터 연결
- testID 기반 탭
- 화면 요소 검색
- 스크린샷
- 앱 로그 수집
